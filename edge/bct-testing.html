BCT Testing (Issue Replication)

This testing article explains the verification and testing steps required before escalating any Billing, Content, or Tech (BCT) issue for further investigation.

Overview

When a subscriber reports an issue to Customer Care, the first step is to identify the correct issue type (Billing, Content, or Tech) so you can determine the appropriate next actions.

  • Billing: Issues related to payment processing, refund errors, entitlement access, whitelisting, or billing failures.
  • Content: Missing episodes, incorrect availability, wrong-title playback, or other content-related errors.
  • Tech: Problems affecting playback, app performance, logins, or streaming functionality.

Once the issue type is confirmed, Customer Care should complete three core tasks in their investigation:

  • Perform basic troubleshooting steps.
  • Test or replicate the issue when possible. If testing is not possible, collect screenshots or video from the subscriber.
  • Collect key information from the subscriber, including ruling out potential blockers (e.g., a VPN for tech issues or a chargeback for billing issues).

All must be completed fully before escalating the ticket further.

⚠️ Attention: The BCT articles (Escalation Workflows, Data Gathering Guide, Troubleshooting Steps, and Testing) do not follow a fixed sequence. Customer Care should start with the article that best fits the situation, as the order may change based on the subscriber’s concern. These resources work together and should be applied thoughtfully for each case.

One of the most important core tasks is issue testing. And don’t worry—you won’t get a fail on this part.

Before any issue is escalated for research, we must complete all relevant tests to confirm the problem. Proper testing ensures we understand the subscriber’s experience, prevents unnecessary escalations, and gives HCL the clear information they need to investigate effectively.

Replicating Issues Using the Lab Devices

When helping a subscriber, Customer Care should run their own replication and testing using any matching devices in the Device Lab. The agent handling the case should complete the testing themselves so they understand the issue fully. Observing, documenting, and noting all test results ensures accurate assessment based on firsthand insight.

Tips for Replicating Issues in the Device Lab

Effective replication helps ensure accurate troubleshooting and consistent results. Before running any tests, keep these quick reminders in mind to maintain reliability and avoid false outcomes.

  • Match the subscriber’s setup as closely as possible: Use the same device model, app version, and login flow they’re using.
  • Check the network connection first: Ensure the device is connected to a stable and correct Wi-Fi network (e.g., AU, US, CA, etc.) to avoid incorrect replication.
  • Start with a clean state: Close any unused apps, clear the cache if needed, and restart the device you're using, if possible, to avoid interference.
  • Follow the exact steps described by the subscriber: Replicate their click path, timing, and actions to catch sequence-related issues.
  • Take note of all observations: Document what worked, what didn’t, error messages, loading behaviors, and response times.
  • Document your test results: Screenshots or short screen recordings are helpful and should always be attached to the agent’s notes. This will allow HCL to validate findings more quickly if the case needs to be escalated.
  • Try at least two attempts: Some issues appear only under certain conditions or on repeat actions.
  • Test variations when needed: If the issue doesn’t appear the first time, adjust small factors (e.g., logging out/in, reinstalling the app).
  • Check device settings: Ensure OS updates, app updates, display settings, and time/date settings match typical subscriber environments.
  • Be mindful of shared devices: When done, always return devices to default state after testing (logged out, app reset if changed, cables reconnected).

⚠️ Attention: If you cannot replicate the issue, we must obtain a video that shows the full sequence or clear images that capture the error or behavior..

Replicating Issues While on Live Messaging

If assisting a subscriber through messaging and you need to run a test:

  • Place the subscriber on hold using Chat-Insert :: Hold to set clear expectations (you can ask the subscriber to wait for more than 6-8 minutes as long as they agree to it).
  • Check in regularly and follow the proper hold procedure.
  • Provide short updates on what you are testing and why. Use Chat-Insert :: Add Hold for these updates.
  • Never leave the subscriber waiting without context or feeling unattended.

Clear communication builds trust and improves the overall support experience.

If the Device Is Not Available in the Lab

If the subscriber is using a device that is not available in the Device Lab, such as Fetch or Foxtel, follow the guidelines below:

  • Ask the subscriber to perform the replication on their end.
  • Request proof of the issue, preferably a video that shows the full sequence or clear images that capture the error or behavior.

More visual evidence leads to a more accurate assessment and a quicker path to resolution. You can never have too much information. 

Device Care and Returning Equipment

After using any device from the Device Lab, including laptops, tablets, iPads, mobile devices, or remotes, agents are expected to:

  • Return all devices promptly after testing.
  • Ensure devices are placed back in their proper location.
  • Make sure the equipment is ready for the next team member who may need it.

Proper device management helps maintain smooth operations and ensures availability for everyone.

Phrasing Guidelines

Did you know that testing and replicating can mean different things across teams and regions? The word replicating does not imply success on its own. It only means you attempted to recreate the issue, and using it incorrectly can confuse HCL or the BritBox Technical teams.

Use the phrasing below when documenting your testing:

  • If reproduced: “Tested and replicated (or reproduced) the error.”
  • If not reproduced: “Tested but could not replicate (or reproduce) the error.”
  • If testing isn’t possible: “Unable to test and replicate the error.”

💡 Tip: If testing is not possible, document the reason in the ticket. For example, you may note that the device is not available in the lab.
